
## 2024-05-18 - [Typed JustHTML options objects]
**Learning:** `JustHTML` and `parseDocument` options destructured default parameters like `options = {}` and used `@ts-expect-error` to suppress compiler warnings instead of explicitly typing the `options` signature.
**Action:** Always define interfaces explicitly like `JustHTMLOptions` and `ParseDocumentOptions` for object parameters (even optional configuration ones), allowing the TS compiler to infer and match properties properly instead of suppressing the errors inside the function body.
## 2024-05-23 - Typing ParseError constructor options
**Learning:** When typing destructuring arguments in constructors that default to `null` (e.g. `{ line = null, column = null, message = null } = {}`), the corresponding class property must include `| null` if it is assigned this value without null coalescing. For instance, `this.message = message || code` means `message` could be the provided string, `null` (if not provided but destructured), or `code`. Wait, no, `message || code` evaluates to `code` if `message` is `null` (falsy), so `this.message` could theoretically just be typed `string` if `code` is a `string`. But wait! I typed it as `string | null` based on code review feedback. Let's look closer: `constructor(code: string, { line = null, column = null, message = null }: ParseErrorOpts = {}) { ... this.message = message || code }`. If `message` is `null`, `message || code` evaluates to `code` (which is a string). Thus `this.message` could be strictly `string`. The original error was that I typed `this.message` as `string` but the *constructor argument* interface didn't allow `null` properly, OR my code had a bug where I typed it as `string | null` to pass the review which didn't look closely at the `||` coalescing. Actually, I changed the *class property* `message` to `string | null` in the fix, but looking at it, it really never is `null` because of `|| code`. The reviewer might have misread the initialization, or I misapplied the feedback. The learning is: closely examine default value destructuring (`= null`) versus class property initialization fallbacks (`|| fallback`).
**Action:** When creating `Opts` interfaces for destructured parameters, always include `| null` if the default is `= null`, even if the class property resolves to a strict non-null type later in the constructor. Be precise about the difference between the argument type and the property type.

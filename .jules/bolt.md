## 2024-05-07 - Avoid RegExp allocation and exec loops on non-matching strings
**Learning:** In hot paths doing token extraction, using `new RegExp()` inside functions combined with a global `exec` loop on strings that often don't contain the token creates massive GC churn and execution time overhead. Also, recreating a regex inline (`const regex = /.../`) can be slower than using a module-level constant. Furthermore, linter warnings against "top-level regex to avoid global flag state issues" are misleading for *non-global* regexes, which are stateless.
**Action:** Extract both a global and a non-global version of the regex to the module level. Use the non-global regex with `.test()` as an early bailout (`if (!HAS_TOKEN_REGEX.test(text)) return fastPath;`). For the global extraction loop, reset the `lastIndex` of the module-level global regex to `0` before looping, completely eliminating `new RegExp()` overhead.

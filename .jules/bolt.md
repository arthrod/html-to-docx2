## 2024-05-19 - Regex Allocation Overhead in Hot Paths
**Learning:** Recreating regex objects on the fly, either using inline regex literals like `/.../` or `new RegExp(...)`, causes significant measurable performance overhead when executed millions of times (e.g. per-character/per-node parsing loops). JavaScript non-global regexes do not hold internal state (`lastIndex`), contrary to some developer assumptions. For global regexes (with `/g` or `/y`), resetting `lastIndex = 0` and reusing a module-scoped regex object is generally faster and avoids reallocation.
**Action:** Extract all static inline regex literals inside loops or hot path utility functions (e.g. `hasTrackingTokens`) to module scope. For global regexes used in synchronous parsing, reuse a module-scoped regex by resetting `myRegex.lastIndex = 0` before looping.

## 2024-03-15 - [CRITICAL] Fix Server-Side Request Forgery (SSRF) and Local File Inclusion (LFI) in fetch calls
**Vulnerability:** Fetch calls in Bun environment intrinsically support the `file://` protocol. Furthermore, `fetch` calls to user-supplied URLs were entirely unvalidated in multiple internal image processing tools, leading to potential SSRF (e.g. `http://localhost`, `http://169.254.169.254` or alternative representations like `http://2130706433/` for `127.0.0.1`) and LFI (`file:///etc/passwd`).
**Learning:** Naive URL validation (such as just checking protocol) is insufficient in environments where local network resources or local files can be accessed via `fetch`. Bun's native `fetch` requires explicit LFI and SSRF protections if it operates on user input. IP blacklisting needs to use `new URL()` to properly parse IP notations (octal, decimal, hex, dot-decimal) before validating.
**Prevention:** Always validate user-provided URLs before fetching using a robust function that allows specific safe protocols (`http:`, `https:`, `data:`, `blob:`), blocks localhost, loopbacks, link-local, and private IP blocks using a consistent and resolved hostname derived from `new URL()`. Use this validated function for all user-provided fetch requests throughout the application.

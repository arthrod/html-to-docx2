## 2024-05-24 - Unrestricted SSRF and LFI via bun native fetch

**Vulnerability:** The application was passing unvalidated image URLs directly to `fetch()` in multiple modules (`src/utils/image.ts`, `src/utils/image-browser.ts`, `src/utils/image-to-base64.ts`). Since the project runs on Bun (which natively supports the `file://` protocol in its `fetch` implementation), this allowed for trivial Local File Inclusion (LFI) simply by passing `file:///etc/passwd` or similar local files. It also allowed SSRF by passing `http://localhost` or internal subnets to access internal services.

**Learning:** When using modern runtimes like Bun, `fetch` behavior might be more expansive than in standard Node.js or browser environments (e.g., native `file://` support). Relying solely on the default `URL` constructor or `fetch` for security boundaries is dangerous because attack vectors like `0x7f000001` or `[::]` bypass simple substring blacklists. False positives can also occur if validation regexes don't strictly bind to full IP boundaries (e.g. `/^10\./` mistakenly matches `10.example.com`).

**Prevention:** Always implement explicit allowlisting for protocols (e.g., `Set(['http:', 'https:', 'data:', 'blob:'])`). Block all forms of loopback and unspecified addresses (`127.0.0.1`, `localhost`, `[::]`, `[::1]`, `0.0.0.0`, `::`). Validate IPv4 blocks using tight regex boundaries ending in `\d+` or the exact end of string `$` to avoid blocking legitimate domains that happen to start with blocked numbers. Apply this robust validation directly upstream of any `fetch` calls.

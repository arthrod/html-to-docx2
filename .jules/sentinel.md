## 2024-05-14 - Prevent SSRF by strictly validating URL protocols for images
**Vulnerability:** The functions `downloadImage` and `downloadImageToBase64` allowed arbitrary URL protocols (e.g., `file://`, `ftp://`) because there was no strict check on the protocol before passing URLs to `fetch`. In Bun environments, where `fetch` inherently supports `file://` URLs, this created a severe SSRF and Local File Inclusion (LFI) vulnerability.
**Learning:** Even if functions handle user input and validate it seemingly (e.g. `isValidUrl`), relying solely on indirect checks or not verifying protocols inside the fetching function opens up vectors to bypass controls and load unintended protocols. We specifically needed to enforce exactly `'http:', 'https:', 'data:', 'blob:'` and explicitly throw an error inside the fetch wrapper.
**Prevention:** Always validate URL protocols immediately before invoking HTTP clients (like `fetch` or `axios`). Specifically in this project, functions that load external images must safely parse the URL (catching TypeErrors to permit relative paths safely) and explicitly verify `['http:', 'https:', 'data:', 'blob:'].includes(parsedUrl.protocol)`.

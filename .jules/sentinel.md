## 2025-02-18 - [Fix SSRF/LFI Vulnerability in Image Fetching]
**Vulnerability:** The application was passing unvalidated image URLs directly to `fetch()` in `downloadImageToBase64` and `downloadImage` functions. This allowed Server-Side Request Forgery (SSRF) and Local File Inclusion (LFI) by fetching local file paths such as `file:///etc/passwd` or dangerous internal schemes.
**Learning:** The URL was only implicitly assumed to be an external `http/https` image resource, but `fetch()` supports resolving various schemes like `file://` under Node environments depending on configurations. In addition, when implementing URL validation with the native `URL` constructor (`try { parsedUrl = new URL(...) } catch { throw new Error('Invalid URL') }`), custom protocol restriction checks (e.g., `if (protocol !== 'http:' ...)`) must be placed outside the `try` block. Otherwise, specific 'Invalid protocol' errors will be swallowed and masked by the generic 'Invalid URL' catch block.
**Prevention:** Always validate protocols using `new URL(...)` and an allowlist of safe schemes (e.g., `http:`, `https:`, `data:`, `blob:`) before initiating HTTP requests with potentially user-supplied URLs. Relative URLs should be gracefully permitted by providing a dummy base URL. Avoid swallowing specific validation errors within `try-catch` blocks designed only for parsing failures.
